AI 스마트검색 아키텍처 — FE에서 외부 RAG까지 추적 (smep-ufe)

대상: smep-ufe의 AI 스마트검색(/req/ai/ai-smart-search)이 화면에서 외부 RAG 서버까지 어떻게 이어지는지 알아야 하는 사람. 동시에 이 문서는 "추적기" 다 — 04(고급 React)·05(브라우저 디버깅)에서 배운 방법을 실전에 적용해, FE 코드만으로 외부 시스템을 어디까지 알 수 있고 어디서 멈추는지(증거의 경계) 를 보여준다. 모든 인용은 실제 코드. 선행: 04-react-advanced.md(커스텀 훅·배럴 파일) · 05-browser-debugging.md(네트워크·스트림 관찰) · 01-url-fe-be-tracing.md(추적 기본기)

0. 먼저 — "검색"이 두 개다 (이름의 함정)

이 시스템엔 이름이 비슷한 검색이 둘이고, 엔진이 완전히 다르다. 헷갈리면 추적이 처음부터 어긋난다.

통합검색 AI 스마트검색 (이 문서)
엔진 smep-be의 Wisenut 상용 검색엔진 외부 AI 서버의 RAG
방식 BM25 + 한국어 형태소분석 = 키워드 검색 벡터검색(Milvus) + 리랭킹 + LLM 요약
경유 smep-be → /api/v1/search/totalSearch FE → 외부 AI 서버 (smep-be 안 거침)
RAG? ❌ ✅

⚠️ "통합검색"이란 이름이 세 군데에 흩어져 있다 — ① smep-be의 Wisenut API, ② FE의 useTotalSearch 훅(죽은 코드, 04 §1-2), ③ AiSmartSearch의 sr-only 라벨. 이름만 따라가면 셋을 같은 것으로 착각한다. 메인 검색창은 화면상 "통합검색"이지만 실제로는 /req/ai/ai-smart-search(RAG)로 이동한다(MainPage.jsx:24). 이 문서가 다루는 건 ✅ 쪽이다.

1. 한눈에 — 4겹 추적 지도

AI 스마트검색은 한 파일이 다 하지 않는다. 책임이 4겹으로 분리돼 있고, 각 겹은 바로 아래 겹에 위임한다.

위→아래로 각 겹이 바로 아래 겹에 위임한다. FE 박스(①~④) 안이 우리가 읽을 수 있는 코드고, 굵은 화살표가 박스를 넘어가는 ④→⑤ 지점부터는 외부 서버(우리 코드 아님)다.

핵심 한 줄: FE의 일은 "질의·필터를 실어 보내고, 서버가 흘려주는 단계별 산출물을 받아 그리는 것"까지다. 검색·추론·생성의 로직 자체는 ⑤(서버)에 있다.

2. 각 겹이 하는 일 (위에서 아래로)

① AiSmartSearch.jsx — 화면·소비

  • 메인 검색창에서 진입(MainPage.jsx:24 navigate('/req/ai/ai-smart-search', { state: { q } })).
  • useProgramSearch()로 검색 상태를 받고, 전체를 <ProgramSearchProvider stream topK rerankerTopK domain groupByField>로 감싼다(AiSmartSearch.jsx:753). 이 props(topK/rerankerTopK)가 RAG 파라미터다.
  • ⚠️ 함정: 같은 파일이 useTotalSearch도 import하지만 호출이 주석 처리된 죽은 코드다(AiSmartSearch.jsx:135,172). 실제 결과는 useProgramSearch에서 온다(04 §1-2).

② useProgramSearch.tsx — SMES 도메인 어댑터

범용 검색 훅을 받아 SMES(지원사업)용으로 가공만 한다. 자체 HTTP는 없다.

// useProgramSearch.tsx:14 — 진짜 엔진은 범용 레이어에서 가져온다
import { useSearchContext as useSDKSearchContext } from "../../../react";
// :42 — 결정적 단서: 필터가 "Milvus 필터"다 (= 벡터DB)
/** Milvus 필터 (7개): regions, companySizes, supportFields, ... */
// search 내부 (화면에서 본 332~337행)
const backendFilters = buildBackendFilters(filters);          // SMES 필터 → Milvus 필터
const effectiveQuery = query.trim() || "지원사업";            // 빈 쿼리 기본값
sdk.search(effectiveQuery, { ...options, filters: backendFilters });  // 위임

어댑터가 더하는 일은 셋뿐: Milvus 필터 변환 · 쿼리 기본값 · 결과 매핑(mapSourceToProgram/parseAISummary).

③ useCubeIAxSearch.ts — 범용 검색 훅

도메인 무관한 검색 상태·스트리밍을 관리한다. 반환 타입(UseCubeIAxSearchReturn)에 RAG 산출물이 그대로 드러난다:

// useCubeIAxSearch.ts (89~120행)
results, total, content, streamingContent,   // 검색결과 + 스트리밍 요약
status,                                        // "검색 중..."/"분석 중..." (StatusEvent)
clarificationMessage, clarificationQuestions,  // 되묻기
followupSuggestions,                           // 후속 질문
rewrittenQuery,                                // 쿼리 재작성
queryAnalysis,                                 // 쿼리 분석
sessionId,                                     // 멀티턴 대화 연결

이 필드 구성이 "검색→분석→재작성→리랭킹→되묻기"의 다단계 파이프라인을 증언한다. 단순 1턴 RAG가 아니다.

④ api/client.ts — 전송(transport) 계층

실제 HTTP·SSE를 담당. 여기가 FE 코드 추적의 마지막 정거장이다.

// client.ts:738 — 검색은 POST {baseUrl}/search (스트리밍)   ※ 채팅은 /chat (:359)
const response = await fetch(`${this.baseUrl}/search`, {
  method: 'POST',
  headers: this.getStreamingHeaders(),
  body: JSON.stringify({
    message, session_id, profile, filters,
    document_context,                       // group_ids (빠른 필터링)
    stream: true,
    top_k: ... ?? DEFAULT_TOP_K,            // 검색 후보 수
    reranker_top_k: ... ?? DEFAULT_RERANKER_TOP_K,   // ★ 리랭킹 단계
    include_citations, group_by_field, domain,
    max_tokens: ...,                        // 생성 길이 (100~2048, 기본 1024)
    ...(agent && { agent }),                // ★ 어느 에이전트로 보낼지
  }),
});

client.ts가 하는 일은 요청 직렬화 + SSE 수신 + abort 처리가 전부다. 프롬프트·도구 정의·오케스트레이션은 한 줄도 없다. 그것들은 ⑤(서버)에 있다.

⑤ 외부 AI 서버 — 경계 너머

baseUrl은 App.jsx에만 있다(:11 prod smes-tipa.go.kr/aiax-dev/v1, :16 dev ax.llmonx.kr:28443/v1). /v1 접미사는 OpenAI 호환 API 컨벤션이지만, agent 같은 커스텀 필드를 얹은 자체 RAG/에이전트 API다. 여기부터는 우리 코드가 아니다.

3. RAG임을 코드에서 읽는 법 — "신호"의 목록

"이게 RAG다"를 막연히가 아니라 코드 단서로 판정하는 법. 두 방향에서 본다.

3-1. 요청측 신호 (FE → 서버로 무엇을 보내나)

신호 위치 의미
top_k / reranker_top_k client.ts:370,371 벡터검색 후보 수 + 리랭킹 단계 존재
Milvus 필터 useProgramSearch.tsx:42 검색 엔진이 벡터DB(Milvus)
domain / group_by_field client.ts:374,375 도메인 라우팅·그룹핑
agent (기본 router agent) core/types.ts:23 서버측 named agent + 라우터 구조
include_citations client.ts:373 출처 인용 = RAG의 근거 제시

3-2. 응답측 신호 — SSE 이벤트 14종 (core/types.ts:310)

서버가 스트리밍으로 흘려보내는 이벤트 타입이 파이프라인 단계 그 자체다:

'session' | 'status'(검색 중·분석 중) | 'sources'(sources-first) | 'token'(스트리밍 텍스트)
| 'citations' | 'clarification'(되묻기) | 'followup'(후속질문) | 'analysis'(분석)
| 'item_details' | 'context' | 'search_result' | 'rewrite'(쿼리 재작성) | 'end' | 'error'

rewrite·analysis·clarification이 독립 이벤트로 존재 = 서버가 쿼리 재작성·분석·되묻기를 단계로 수행하고 실시간 보고한다(고급/agentic RAG의 모습). sources(검색결과)를 먼저 보내고 token(요약)을 뒤이어 스트리밍하는 "sources-first" 패턴도 전형적 RAG UX다.

05 디버깅과 연결: 이 이벤트들을 눈으로 보려면 Network 탭에서 /search 요청의 EventStream 탭을 열면 status→sources→token…이 실시간으로 찍힌다. 글로 읽은 14종이 거기서 살아 움직인다.

4. 클래식 RAG vs 에이전트형 — 그리고 "증거의 경계"

여기가 이 문서의 핵심이자, FE 추적이 가르쳐주는 가장 중요한 교훈이다.

4-1. 두 패턴

클래식 RAG (retrieve-then-generate) 에이전트형 (agentic / tool-calling)
검색 주도 앱/서버가 먼저 검색 → LLM 1회 생성 모델이 도구를 스스로 호출 → 추론 → 재검색 (루프)
모델 역할 컨텍스트로 생성만 검색을 오케스트레이션

4-2. 우리가 확증한 것

  • ✅ AI 스마트검색은 RAG가 맞다 (Milvus·리랭킹·LLM 스트리밍 요약).
  • ✅ 단순 1턴이 아니다 — rewrite/analysis/clarification/followup 단계가 있는 다단계 파이프라인.
  • ✅ 서버는 named agent + router 구조 (agent 기본값 "router agent", 주석 "RAG agent" — core/types.ts:23,352).

4-3. 우리가 확증할 수 없는 것 (그리고 그게 정상인 이유)

"하나의 모델을 tool-use로 에이전트화한 것인지, 여러 전문 모델(쿼리분석·임베딩·리랭커·생성 LLM)의 파이프라인인지"는 FE 코드로 알 수 없다. 근거:

  • SSE 이벤트에 tool_call/tool_result가 없다(§3-2 목록). 서버는 모델의 내부 도구 호출을 감추고, 사용자에게 보여줄 단계 상태와 결과만 추상화해 내려준다.
  • agent는 서버측 논리 단위의 이름표일 뿐 — FE는 그 이름을 실어 보낼 뿐, 그 안의 모델 구성을 모른다.
  • client.ts엔 프롬프트·도구 정의가 한 줄도 없다(§2-④). 그것들은 서버 소스에 있다.

→ 이 "빈자리"는 누락이 아니라 의도된 설계다:

  • 보안 — 프롬프트·도구 정의·모델 키가 FE 번들에 박히면 누구나 본다. 서버에 둬야 한다.
  • 유연성 — 프롬프트/에이전트 구성을 바꿔도 FE 재배포가 불필요.

정직한 미결로 남긴다: 다중 모델 vs 단일 tool-use 에이전트 — FE 코드로는 미확정. 확답은 외부 AI 서버 레포(우리 손 밖)에 있다. 단 일반론으로, "리랭킹"이 있으면 보통 최소 2모델(임베딩+리랭커)에 생성 LLM까지 관여하므로 순수 단일 모델일 가능성은 낮다 — 그러나 이는 통념이지 이 서버의 증거가 아니다.

5. 추적 레시피 (다른 외부 연동에도 재사용)

이번에 쓴 방법을 일반화한다. 외부 시스템에 붙는 어떤 FE 코드든 같은 순서로 판다.

  1. 소비처에서 시작 — 화면 컴포넌트가 import하는 훅/모듈을 찾는다(AiSmartSearch.jsx의 useProgramSearch).
  2. 배럴 파일에서 한 겹 더 — index.ts가 export … from "./x"만 있으면 거긴 문패다. "선언으로 이동"을 한 번 더 눌러 본체로(04 §1, 01 기법). program/index.ts → hooks/useProgramSearch.tsx.
  3. 계층을 구분하라 — 도메인 어댑터(②) / 범용 훅(③) / 전송(④)은 역할이 다르다. "이 파일은 가공인가, 상태관리인가, HTTP인가"를 매번 물어라.
  4. transport에서 멈추는 지점을 인식 — fetch/SSE만 있고 비즈니스 로직이 없으면 거기가 FE의 끝이다. 그 너머는 서버.
  5. 요청 바디 + 이벤트 타입으로 서버 능력을 추론 — 보내는 필드(top_k/agent…)와 받는 이벤트(rewrite/status…)가 서버가 무엇을 하는지 알려주는 거의 유일한 FE측 증거다. 단, 추론과 확증을 구분하라.

6. 자주 막히는 지점 (체크리스트)

  • "통합검색이 RAG인 줄 알았다" → 아니다. 통합검색=Wisenut 키워드(smep-be), RAG=AI 스마트검색(외부). 이름이 겹칠 뿐(§0).
  • useTotalSearch를 고쳐도 AI 검색이 안 바뀜 → 그건 죽은 코드. 실제 경로는 useProgramSearch(§2-②, 04 §1-2).
  • "프롬프트가 코드에 안 보인다" → 정상. 프롬프트·도구는 서버에 있다(§4-3). FE엔 요청/응답만.
  • 결과가 단계적으로 뜨는 이유 → SSE 이벤트(sources→token)를 순서대로 받기 때문. 05의 EventStream 탭으로 확인(§3-2).
  • 응답이 비어 보임 → status/error 이벤트부터 확인. 서버가 clarification(되묻기)을 보냈을 수도.
